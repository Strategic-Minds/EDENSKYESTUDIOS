# Eden Skye Studios Governance And Safety

## Governance Objective

Let Eden Skye Studios move fast without losing control, brand safety, legal safety, platform safety, or business authority.

## Core Rules

1. Eden Skye is fictional and adult.
2. Do not claim Eden is a real human person.
3. Do not imitate or clone real people without lawful grounded source and permission.
4. Keep all media platform-safe and non-explicit.
5. Keep all public, commerce, payment, deploy, and schema actions approval-gated.
6. Log receipts for every meaningful automated action.
7. Preserve Drive/Git source truth.
8. Treat external tools as execution layers, not governance authorities.

## Protected Actions

These actions require explicit current-session approval:

- production deploy
- public publishing
- public scheduling
- social account mutation
- Shopify mutation
- Stripe/payment mutation
- Supabase schema/data mutation
- Vercel env/domain mutation
- HeyGen avatar creation/training
- paid media activation
- customer email send
- deleting/moving source assets
- changing finalization/source-of-truth files

## Content Safety

Allowed brand direction:

- luxury editorial
- tasteful glamour
- cinematic femininity
- implied sensuality
- soft confidence
- premium self-growth
- emotionally grounded intimacy

Disallowed:

- explicit sexual content
- graphic nudity
- pornographic framing
- minors or age ambiguity
- non-consensual themes
- exploitative sexuality
- deception about personhood
- real celebrity/person imitation

## AI Avatar Governance

Before any avatar creation/training:

1. Select avatar ID.
2. Confirm source image IDs.
3. Confirm fictional identity boundary.
4. Confirm use case.
5. Confirm consent/training status where relevant.
6. Create approval request.
7. Log receipt.

## Brand Canon Changes

Changes to Eden's face, voice, identity, values, content pillars, or source-image canon require an amendment note.

Amendment note must include:

- changed field
- old value
- new value
- reason
- approval state
- effective date

## Incident Handling

If unsafe or off-brand output is generated:

1. Mark rejected.
2. Save rejection reason.
3. Do not publish.
4. Add to drift notes.
5. Update prompt avoidance rules if pattern repeats.

## Minimum Receipt Standard

Every controlled autonomous run must record:

- timestamp
- operator/tool
- run mode
- source files read
- action attempted
- result
- approval status
- next action
